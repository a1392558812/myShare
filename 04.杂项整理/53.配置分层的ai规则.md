# 配置分层的ai规则

> 项目级 AI Rules，：统一规范 AI 生成代码、文档、脚本的行为

## 为什么需要全局ai规则

- 每个人写 Prompt 标准不一致，AI 输出风格混乱
- AI 乱选技术栈、乱建目录、违反项目架构
- 每次提问重复交代项目规范，重复造指令
- 安全边界失控（生成密钥、跨层调用、废弃 API）

## 注意下什么

- 规则写得过多、模糊
  - 禁止写空泛描述如「写干净代码」，必须具象：组件命名PascalCase，工具函数camelCase；指令过多会降低模型遵循度。
- 重复 Lint 格式化规则
  - 缩进、引号交给 Prettier，AI 规则只做架构、业务边界约束，减少冲突。
- 无版本管理，本地私改规则
  - 所有规则文件必须提交 Git，禁止个人本地新建私有的规则文件覆盖项目标准。
- 缺少优先级定义
  - AGENTS.md > .ai-rules 细分规则 > 个人全局 AI 配置，规则冲突时以项目根 AGENTS.md 为准。
- 不做灰度验证直接全量上线
  - 新规则先小范围测试，收集 AI 违规输出案例迭代优化后再铺开。

## 约束边界

- 交给 ESLint/Prettier 处理：缩进、引号、分号、unused 变量、命名；不要写入 AI Rules，避免冗余冲突；
- AI Rules 只写 Lint 无法强制的内容：架构分层、目录规范、依赖选型、安全边界、脚本白名单、输出格式、Skill 调度逻辑。
- AGENTS.md 控制在 200 行以内；
- 细分规则单文件≤100 行，长内容拆分为多个文件；
- 每条规则遵循：约束内容 + 原因 + 正反示例，AI 更容易遵循。
- 规则修改走 PR，由技术负责人评审；
- 修改说明：变更规则、解决什么场景问题；
- 上线灰度：先 1-2 人试用 2 天，无大量违规生成再全团队推广。
- AI 生成代码后自动执行pnpm lint拦截违规代码；
- 复杂架构新增脚本校验目录分层；
- 提交 PR 时自动读取 AGENTS.md，校验代码是否符合 AI 规则。

## token优化

### Token计费逻辑

- 输入 Token（prompt）：你发给模型的所有内容（系统提示词、历史对话、项目规则、文件上下文、用户提问）都计费；
- 输出 Token（completion）：模型返回的回答、代码、文本计费；
- 长上下文模型（128k/256k）单价更高，长文本不做裁剪成本翻倍；
- 项目级 AI Rules、Skill、长文档全部计入输入Token

### 精简Token

- 避免大段冗余描述、重复 ESLint/Prettier 格式化规则、无边界废话、大量示例堆砌，剥离可由 Lint 兜底的规则，缩进、引号、分号、变量校验全部交给 ESLint，AI Rules 只保留架构分层、依赖约束、安全红线、输出格式。
- 规则短句化，删除修饰词，避免废话，比如：“请你务必严格遵循本项目目录架构，不可以跨层导入，不然代码会难以维护”，改为：‘除/utils下工具函数与/components公共组件，禁止跨层导入’
- 分阶段动态加载规则，下文详述
  - 写代码 → 只加载 code-style + architecture
  - 查安全问题 → 只加载 security
  - Git 操作 → 只加载 git-workflow

### 设置API 参数限制 max_tokens

> 当确实需要长上下文交互，则需要对历史对话摘要压缩，当历史轮次过多时，不直接删掉，调用一次低成本小模型把多轮对话浓缩成百字摘要，替换原始完整历史

// 代码小修改：max_tokens=1024
// 页面完整组件：max_tokens=4096
// 简单问答：max_tokens=512

## 主控规则

> 自动前置加载，定义角色、全局硬性约束、读取子规则逻辑，控制上下文体量，在 200 行内，避免指令冲突、Token 爆炸

```markdown
# AGENTS.md 项目级AI总规则
## 1. 身份定位
你是本Vue3+TS前端项目专属开发Agent，仅基于项目现有技术栈输出代码，禁止引入未安装第三方库。

## 2. 全局强制约束（最高优先级，不可违反）
1. 所有代码缩进2空格，遵循项目prettier/eslint配置，不生成4空格缩进代码；
2. 优先复用项目现有通用组件、工具函数，禁止重复封装基础能力；
3. 分层严格：views仅做页面渲染，api统一放在src/service，utils存放纯工具；
4. 不生成硬编码密钥、token、线上域名，所有环境变量从.env读取；
5. 输出代码必须带完整TS类型定义，禁用any（特殊场景需标注注释说明）；
6. 输出完成后自动提示：执行`pnpm lint`校验，`pnpm dev`本地验证。

## 3. 子规则读取逻辑
执行不同任务时自动读取对应细分规则：
- 架构/目录修改 → 读取 .ai-rules/01-architecture.md
- 写组件/工具函数 → 读取 .ai-rules/02-code-style.md
- 接口、鉴权、隐私逻辑 → 读取 .ai-rules/03-security.md
- Git提交、PR评审 → 读取 .ai-rules/04-git-workflow.md
- 脚本、打包、格式化命令 → 读取 .ai-rules/05-cli-command.md
- 复杂业务流程 → 读取 .ai-rules/06-skills-index.md 调用内置Skill

## 4. 输出格式规范
1. 代码块完整可直接复制，不省略import、不遗漏类型；
2. 复杂逻辑分步骤说明，先给出完整代码，再讲改动点；
3. 若规则存在冲突，以本AGENTS.md优先级最高；
4. 发现自己生成代码违反规则时，主动自查修正并标注问题点。

## 5. 禁止行为
- 新增未沟通的第三方依赖、UI库；
- 跨层直接导入（views直接import接口、组件耦合业务逻辑）；
- 生成包含明文密码、密钥、敏感用户数据代码；
- 修改dist、node_modules、.env.production文件；
- 使用npm/yarn，仅允许pnpm脚本；
- 注释掉eslint校验规避规范。
```

```markdown
# 架构与目录分层规则
## 技术栈锁定
Vue3 + <script setup> + TypeScript + Vite + Pinia + VueRouter4 + Element Plus
禁止：Vue2、Options API、class组件、jQuery、原生axios（统一封装request）

## src目录强制结构
src/
├─ api/       接口请求、请求拦截器、类型定义
├─ components/全局公共组件
├─ views/     页面业务组件（禁止存放工具/请求）
├─ store/     pinia状态管理
├─ utils/     纯工具函数（颜色转换、数组处理、日期等）
├─ hooks/     组合式复用逻辑（团队Skill存放处）
├─ assets/    静态资源、样式
└─ router/    路由配置

## 分层调用约束
1. views → 仅允许导入 components / hooks / store
2. hooks → 可导入 api / utils，禁止直接操作DOM
3. api → 仅处理http请求，无业务判断逻辑
4. utils → 纯无副作用函数，禁止导入store/api
```
