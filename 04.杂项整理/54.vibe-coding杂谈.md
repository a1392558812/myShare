# vibe coding杂谈

## 概念

其实特简单，也没啥规范不规范的，就是指 `人类用大白话给AI，让Ai去干活、写代码、把东西整出来`。

Ai前时代，编程就是「人精准的敲代码」，人类主导代码结构跟质量，Ai顶多联想点api/工具函数啥的，后来Ai功能越来越猛，慢慢的能预测代码结构了，在那个时间段，我经常是先写一段注释，比如“写个函数拉取用户数据”，然后Ai就把结构给你生出来，我按个Tab接受补全，再自己上去修补修补，这其实就已经有vibe coding那味儿了，只是当时没人起这名字罢了，如果你当时足够知名，你都可以给他命名为“牢ai”。

现在彻底变，人定需求，Ai落地，人做决策、Ai做执行，开发者的核心能力从「写得出代码」变成了「说得清需求，辩得对对错，调得好细节」

## 是什么

1. **弱化手写，强化沟通**  
   绝大多数代码由 AI 生成，开发者极少逐行敲代码，更多时间是写提示词、改需求、提优化指令、排查报错。编程从「体力活」变成了「沟通活」。

2. **先跑起来，再做优化**  
   摒弃传统开发那种层层设计、层层校验的严谨流程，核心逻辑是「快速落地、快速试错」。先让功能跑通，再逐步优化性能、修复bug、规范代码、重构架构。

3. **模糊细节，掌控全局**  
   开发不需要记住所有语法、所有工具细节、所有兼容规则，知道有哪个个api，有哪个功能，有哪个第三方库能实现什么就行，把琐碎、重复、机械的编码工作交给 AI，自己专注把控业务逻辑、产品体验、整体架构方向。

4. **迭代式纠错，而非一次性写对**  
   传统开发追求一次写对、少出bug，。Vibe-Coding接受「初始代码不完美」，通过报错反馈、效果预览、需求微调，一轮轮迭代修复问题，适配快速开发的节奏。给ai擦屁股。

## 改变了什么

开发效率的升级

对独立开发者、个人开发者、初创团队来说，传统开发的成本太高：前后端协作问题，多端设备问题等等，大量时间消耗在重复劳动上，真正的业务创新时间被严重挤压。

而 Vibe-Coding 完美解决了这个痛点：AI 包揽所有重复、繁琐、模板化的工作，开发者只需要聚焦核心创意和业务逻辑。以前几天才能做完的小功能、小工具、脚本、插件，现在十几分钟就能落地，而且对于像我这种独立开发者，很多demo和工具脚本压根没有迭代需求，将就用能出东西就行，不需要传统开发那么严谨一套，产品是我，用户也是我。

哪怕是语法不熟、框架不精通、久未写代码的人，只要能清晰描述需求，就能靠 AI 完成开发。说破大天，能跑就是好活👍

## 争议

1. **代码质量不可控，隐性坑极多**  
   AI 生成的代码普遍存在冗余（特别多且不合理的空值兼容写法，有时候反而不好排查报错）
   不规范（一个文件干几千行，还得人工指导拆分出哪些模块）
   兼容差（臆想一些api，虚空捏造一些设备漏洞，需要兼容，正儿八经的漏洞一声不吭，该防的不防，不该防的瞎防）
   逻辑漏洞（老需求要么忘了，要么就是一个废弃的需求，Ai念念不忘，删都删不干净）
   内存泄漏（边界隐患，特别是并发与内存回收之类的）
   安全隐患（Ai权限过高，主要体现在一些py/js脚本使用一些类似于删除/操作系统命令等等）。
   很多bug不会立刻暴露，会在后续迭代、上线、扩容时集中爆发，长期纯靠Vibe开发，会产出大量「屎山代码」，

2. **开发者能力退化**  
   长期不手写核心代码，在写一些基础工程时，会忘东西，刚开始还会尝试给Ai擦屁股，慢慢的演化为让ai给Ai擦屁股，退化自己的排错能力，
   在架构思维上，只会提需求、不愿意再触及底层和读源码、不会重构代码，一旦Ai生成异常，就完全无从下手，

3. **无规范、无设计、无沉淀**  
   纯氛围开发容易陷入「瞎迭代」，没有统一规范/架构/文档，项目越写越乱，后期迭代成本远远高于传统开发。其实，目标人群与主力人群有很大的错位，这种野蛮式开发比较适合我这种个人开发者，上文有阐述，但团队协作就是另外件事

4. **复杂项目完全不适用**  
   大型项目、高并发服务、金融/医疗等严谨业务、底层框架开发，绝对不能靠「感觉开发」，必须严谨设计、精准编码、全面测试，Vibe-Coding 的试错模式完全无法适配。

## 平衡

说到底，Vibe-Coding本身没有对错，错的是完全依赖、无脑摆烂、放弃思考的开发心态。工具是死的，人是活的,

真正成熟的Ai开发模式，从来不是纯氛围编程，而是 「严谨架构 + 氛围迭代」 。手工匠心和ai流水线，不能一锅端。

- 适合场景：快速原型、小工具、脚本、插件、个人项目、临时需求、页面样式、通用功能开发，用Ai提速、解放双手大胆用，
- 严谨落地的场景：核心业务逻辑、底层架构、数据安全、性能优化、线上稳定服务，必须自己把控、逐行审核、深度优化，绝不全权交给Ai。

## 规范

- 规范前置，x需求澄清，写清用户、场景、约束、验收，比如产品需求，eslint设置，团队级别rule等
- 架构设计，明确ai是否可以安装第三方包，已经给ai喂当前使用第三窟的文档，防止ai幻觉自己臆想api
- skill筛选好，自己炼/公司炼/社区炼的，看自己需求
- 前置时多使用ai的ask模式，先确认要改动/操刀的地方
- 决定开发路线是“先实现功能跑起来再迭代”还是“先拆任务，小步改代码+验收”，
- 验收，一定要给ai擦屁股，

### 需求文档

> ai每次实现功能需要落地文档，并完善文档

```markdown
## 核心属性系统

### 属性点与属性值区分

- **属性点**：玩家通过升级、自由分配获得的点数
- **属性值**：属性点×系数得出的最终属性数值
- **装备基础属性**：存属性点（需要乘以系数转为属性值）
- **装备基础词条**：存属性点（需要乘以系数转为属性值）
- **装备强力词条**：存属性值（已经乘过系数，直接相加）

### 属性计算公式

| 属性     | 计算公式                                                                     |
| -------- | ---------------------------------------------------------------------------- |
| 物理攻击 | 基础物攻值(55) + 玩家物攻点×1.5 + 装备物攻点×1.5 + 装备强力物攻值            |
| 法术攻击 | 基础法攻值(10) + 玩家法攻点×1.1 + 装备法攻点×1.1 + 装备强力法攻值            |
| 防御     | 基础防御值(5) + 玩家防御点×2 + 装备防御点×2 + 装备强力防御值                 |
| 速度     | 基础速度值(5) + 玩家速度点×1 + 装备速度点×1 + 装备强力速度值                 |
| 生命上限 | 基础生命上限值(550) + 玩家生命上限点×10 + 装备生命上限点×10 + 装备强力生命值 |
| 法力上限 | 基础法力上限值(100) + 玩家法攻点×1.1 + 装备法攻点×1.1 + 装备强力法力值       |

> **注意**：法力上限不依赖独立的属性点，而是通过法攻点自动增加

### 战斗属性（不可加点）

| 属性         | 玩家基础值 | 敌人范围 | 说明                 |
| ------------ | ---------- | -------- | -------------------- |
| 暴击率       | 5%         | 0 ~ 30%  | 实际暴击率受装备加成 |
| 连击率       | 1%         | 0 ~ 30%  | 实际连击率受装备加成 |
| 最大连击次数 | 1次        | 1 ~ 3次  | 实际值受装备加成     |
| 障碍抗性     | 0%         | 0 ~ 30%  | 抵抗障碍技能的概率   |
| 忽视障碍异常 | 0%         | 0 ~ 30%  | 增加障碍技能成功率   |

#### 暴击计算

- **实际暴击率** = 玩家暴击率 + 所有装备强力词条暴击率（受最大暴击率上限限制）
- **暴击伤害** = 基础伤害 × 暴击倍数(1.5)
```
